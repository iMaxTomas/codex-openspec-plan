## Context

此前已经确认一个稳定现象：在解释 `Codex -> wrapper -> artifact -> Langfuse -> 后续复用` 这类链路时，先讲“真相层 / 观测层 / 索引层 / recall / continuity”会导致理解负担明显增加；而先讲第一人称动作链，例如“我先运行 wrapper，wrapper 再启动 codex，结束后 wrapper 保存结果”，能够显著降低理解门槛。

这说明问题不主要在术语本身，而在解释顺序。对本机 Codex 的长期配置而言，这是一类跨任务稳定成立的高价值规则，适合作为独立规则对象长期保留。

同时，这组 rules 不应混入 `add-minimal-context-and-curated-skills-governance` 的治理正文。后者负责边界；本 change 负责内容。

## Goals / Non-Goals

**Goals:**
- 定义复杂系统解释的默认顺序
- 定义应优先使用的语言风格和结构
- 主动约束容易引发误解的表达方式
- 产出一份可直接放入 Codex 长期配置或 rules 文件的草案

**Non-Goals:**
- 本 change 不直接修改现有 `AGENTS.md`、`CLAUDE.md` 或其他 root 配置
- 本 change 不讨论 artifact、Langfuse、RecallNest 的技术实现细节
- 本 change 不处理某个单一项目的 task-scoped 临时解释文本
- 本 change 不替代治理 change 的边界判断

## Decisions

### 1. 解释复杂链路时必须先给第一人称动作链
默认先解释：

`我先做什么 -> 谁先收到 -> 谁再启动谁 -> 结果存到哪里 -> 我下次怎么继续用`

替代方案：
- 先讲系统分层或架构图  
  放弃原因：会让理解在动作链建立前就进入抽象跳跃。

### 2. 必须主动排除错误链路
在第一轮解释里，必须明确说明哪些理解是错误的，例如：
- 不是内部自动插件调用
- 不是套娃式再次启动第二个同类进程
- 不是当前自动把历史数据回灌给正在运行的进程

替代方案：
- 等用户误解后再纠正  
  放弃原因：会显著增加往返成本，并固化错误心智模型。

### 3. 系统流转应在动作链之后展开
当用户已经看懂“我怎么做”之后，再解释：
- wrapper 做什么
- artifact 存什么
- Langfuse 存什么
- 下一次怎么复用

替代方案：
- 在第一轮解释中同时讲动作链与系统分工  
  放弃原因：信息密度过高，容易打断建立中的动作理解。

### 4. 架构语言必须最后才展开
只有在前两层都成立后，才允许引入：
- 观测层
- 真相层
- 元数据层
- recall / continuity
- control plane

替代方案：
- 默认使用架构术语  
  放弃原因：会让系统说明听起来正确，但难以真正被使用者吸收。

### 5. ASCII 图必须服务于动作理解，而不是只服务于架构展示
ASCII 图优先展示：
- 我
- wrapper
- codex
- 保存
- 再次运行

而不是一上来铺满抽象分层。

替代方案：
- 默认先画总架构图  
  放弃原因：会增强结构感，但不一定增强可操作理解。

## Risks / Trade-offs

- [风险] 规则过度偏向某一种理解习惯 -> 缓解：保留架构层说明，但推迟到后面展开
- [风险] 解释过于口语化导致信息压缩不足 -> 缓解：第一轮先保动作链，第二轮再补系统分工
- [风险] 长期配置里加入过多解释类规则 -> 缓解：只保留这组跨任务稳定成立的高价值解释顺序约束

## Migration Plan

2. 形成一份可直接部署的 rules 草案
3. 明确这组 rules 放入长期配置时的边界说明
4. 为后续 Codex 配置部署 change 提供引用入口
